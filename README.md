# NixOS configuration — roach + crafty

A single flake defining multiple NixOS hosts. Started as roach-only; now
unified so new hosts (and eventual server migrations) share common config
instead of re-solving the same boilerplate each time.

**Note:** this repo is the source of truth for `roach`. `crafty` still runs
its own independent config/repo until it's explicitly synced over and
rebuilt from here — see "Syncing crafty" below.

## Layout

```
/etc/nixos/
├── flake.nix / flake.lock     # inputs + nixosConfigurations.{roach,crafty}
├── .sops.yaml                 # sops-nix recipient/creation rules
├── modules/
│   └── common.nix             # settings identical across every host
└── hosts/
    ├── roach/
    │   ├── configuration.nix          # roach-only config
    │   └── hardware-configuration.nix # autogenerated, machine-specific
    └── crafty/
        ├── configuration.nix          # crafty-only config
        ├── hardware-configuration.nix # autogenerated, machine-specific
        └── secrets.yaml                # sops-encrypted, safe to commit
```

`hardware-configuration.nix` files are autogenerated by
`nixos-generate-config` and shouldn't be hand-edited.

## Hosts

- **roach** — Lenovo ThinkPad X1 Carbon Gen 9, COSMIC desktop, Robert's
  laptop. Users `rc` (admin) and `ac`. Tailscale with nftables as the
  firewall backend. Intel VA-API for hardware video acceleration.
- **crafty** — homelab server. Minecraft (vanilla, via `nix-minecraft`), a
  Home Assistant OS VM (bridged on a dedicated VLAN), Prometheus node
  exporter, and `hermes-agent`.

Settings identical across hosts today (nix flakes setting, locale, timezone,
xkb layout, bootloader, sudo, NetworkManager, Tailscale, the base `rc` user,
and a base package set) live in `modules/common.nix`. Everything else stays
in each host's own `configuration.nix` — deliberately not forced together
where hosts differ for real operational reasons (e.g. roach's nftables/
firewall-trust setup for Tailscale vs. crafty's simpler config).

## Secrets (sops-nix)

Secrets are encrypted with [sops](https://github.com/getsops/sops) using
age identities, wired up via `sops-nix`. Each host's secrets file has two
recipients: a personal admin key and that host's own key, so Robert can
always edit from roach and the host can always decrypt itself at boot.

- **Host decryption**: sops-nix uses each host's existing
  `/etc/ssh/ssh_host_ed25519_key` to decrypt its own secrets automatically —
  no separate key management on the host itself. (roach doesn't run sshd
  today, so it has no host key yet; it doesn't need one until it has its own
  secrets file.)
- **Admin decryption**: Robert's personal `~/.ssh/id_ed25519` (backed up in
  Vaultwarden), converted to an age identity with `ssh-to-age`. The derived
  private key lives at `~/.config/sops/age/keys.txt` on any machine acting
  as an admin editing station — regenerate it any time with:
  ```sh
  ssh-to-age -private-key -i ~/.ssh/id_ed25519 -o ~/.config/sops/age/keys.txt
  ```
  Nothing secret needs to be copied between machines; restoring the SSH key
  from Vaultwarden and re-running that command is enough.

To edit a host's secrets:

```sh
sops hosts/crafty/secrets.yaml
```

To add a new host's secrets for the first time: derive its host age key
(`ssh-to-age -i /etc/ssh/ssh_host_ed25519_key.pub`), add it to `.sops.yaml`
alongside the admin key under a new `creation_rules` entry, then run
`sops hosts/<name>/secrets.yaml` to create and populate it.

`hosts/crafty/secrets.yaml` currently holds:
- `minecraft_env` — `RCON_PASSWORD`, substituted into the Minecraft server's
  `rcon.password` via nix-minecraft's `@VAR@` substitution, so the real
  value never lands in the Nix store.

hermes-agent doesn't have a secret yet — it's authenticated via a Copilot
login rather than an API key, so `services.hermes-agent.environmentFiles`
stays commented out in `hosts/crafty/configuration.nix` until that changes.

## Syncing crafty

crafty's live system still runs its own separate `/etc/nixos` git repo. This
repo's `hosts/crafty/*` is a snapshot built from what was actually on crafty
at the time — it will drift from crafty's live config until it's explicitly
synced over and rebuilt there. That's a deliberate, separate step, not done
automatically.

## Applying changes

```sh
sudo nixos-rebuild switch --flake /etc/nixos#roach
```

To check the flake evaluates without building/switching (covers every host
defined in the flake, not just the local machine):

```sh
sudo nix flake check --no-build
```
